Approximating Derivatives with Finite Difference Methods

Suppose we have a real-valued function of a single variable f(x)f(x)f(x), which we can evaluate for any value of xxx. Suppose we want to compute the derivative of fff at a point x0x_0x0​, but we don't know a general formula for f′(x)f'(x)f′(x). We can approximate the derivative with the following finite difference method:

f′(x0)=f(x0+h)−f(x0)h+O(h). f'(x_0) = \frac{f(x_0+h) - f(x_0)}{h} + \mathcal{O}(h). f′(x0​)=hf(x0​+h)−f(x0​)​+O(h).

If we ignore the effects of floating point precision, we would choose hhh to be as small as possible. Indeed, analytically we have

f′(x0)=lim⁡h→0f(x0+h)−f(x0)h. f'(x_0) = \lim_{h \rightarrow 0} \frac{f(x_0+h) - f(x_0)}{h}. f′(x0​)=h→0lim​hf(x0​+h)−f(x0​)​.

Floating Point Issues

Let's try it for f(x)=e2x,f(x) = e^{2x},f(x)=e2x, x0=1x_0 = 1x0​=1. In 64-bit floating point precision, the smallest positive number we can represent is 5e-324.

f(x) = exp(2*x)
x0 = 1.0
h = 5e-324
f_derivative_approximation = (f(x0+h) - f(x0))/h
@show f_derivative_approximation
f_derivative_approximation = 0.0

What gives? We know that f′(x)=2e2xf'(x) = 2e^{2x}f′(x)=2e2x, so we should have f′(1)=2e2=14.7781121978613f'(1) = 2e^2 = 14.7781121978613f′(1)=2e2=14.7781121978613, but we got 000. Well, let's see what happens when we add x0x_0x0​ and hhh.

x0 = 1.0
h = 5e-324
@show x0 + h
x0 + h = 1.0

We can't represent the number 1+5×10−3241 + 5\times 10^{-324}1+5×10−324 in 64-bit floating point representation. Instead, the value gets rounded off to the nearest number, which is just 111. So in our computation we have f(1+h)=f(1)f(1+h) = f(1)f(1+h)=f(1), so that f(1+h)−f(1)=0f(1+h) - f(1) = 0f(1+h)−f(1)=0, and therefore we get f′(1)=0f'(1) = 0f′(1)=0. In fact, we would get f′(1)=0f'(1) = 0f′(1)=0 for any function f(x)f(x)f(x).

To fix this, let's try the smallest value of hhh for which x0+h≠x0x_0+h \neq x_0x0​+h=x0​ in the computer. The function eps(T) gives the difference between 1 and the next largest number of type T.

h = eps(Float64)
@show h
@show 1+h
h = 2.220446049250313e-16
1 + h = 1.0000000000000002

We now have 1+h≠11+h \neq 11+h=1. Does this fix our issue and give us an accurate approximation of f′(1)f'(1)f′(1)? Not quite. 

f(x) = exp(2*x)
x0 = 1.0
h = eps(Float64)
f_derivative_approximation = (f(x0+h) - f(x0))/h
f_derivative_analytic = 2*exp(2*x0)
@show h
@show 1+h
@show f_derivative_approximation
@show f_derivative_analytic
h = 2.220446049250313e-16
1 + h = 1.0000000000000002
f_derivative_approximation = 12.0
f_derivative_analytic = 14.7781121978613

Choosing an Ideal hhh

We are in the ballpark of the analytic value of f′(1)f'(1)f′(1), but our approximation is off by quite a bit (we don't even have 2 digits of accuracy). What went wrong? Well, just as 1+5×10−3241+5\times 10^{-324}1+5×10−324 is rounded off to 111, each of the floating point operations involved in computing (f(x0+h) - f(x0))/h might have some small floating point errors. But even a small floating point error can result in a large loss of relative precision.

We saw above that machine precision is ϵM≈2×10−16\epsilon_M \approx 2 \times 10^{-16}ϵM​≈2×10−16. Roughly speaking, this means we have about 16 digits of precision in any number we represent. Consequently, any number ccc which we represent may actually be off by an amount around c×ϵMc\times\epsilon_Mc×ϵM​. We can observe this by increasing the precision of our floating point numbers:

using Quadmath
@show Float64(1.1)
@show Float128(Float64(1.1))
Float64(1.1) = 1.1
Float128(Float64(1.1)) = 1.10000000000000008881784197001252323e+00

We see that the 64-bit version of 1.11.11.1 actually stores a number closer to 1.1+8×10−171.1 + 8\times 10^{-17}1.1+8×10−17. This may not seem like a very big deal since the error is very small. But one case where the roundoff error becomes a big deal is in the subtraction of nearly equal numbers. Consider the two nearly equal numbers a=3.3+3×10−16a = 3.3 + 3\times 10^{-16}a=3.3+3×10−16 and b=3.3b = 3.3b=3.3. Analytically, we should have a−b=3×10−16a - b = 3\times 10^{-16}a−b=3×10−16. But let's try performing that computation with 64-bit floating point numbers:

a = 3.3000000000000003
b = 3.3
@show a-b
a - b = 4.440892098500626e-16

We expected a−b=3×10−16a-b = 3\times 10^{-16}a−b=3×10−16, but we got a−b≈4.4×10−16a-b \approx 4.4 \times 10^{-16}a−b≈4.4×10−16. Although the error is small compared to aaa and bbb, it is significant compared to the analytic value of a−ba-ba−b. We went from having 16 digits of precision in aaa and bbb to having not even one digit of precision in a−ba-ba−b. This may seem like an extreme example, but we can see similar effects for something less extreme:

a = 3.30003
b = 3.3
@show a-b

a - b = 3.0000000000196536e-5
aaa and bbb are no longer quite so close, and we have 11 digits of precision in a−ba-ba−b. There is still a noticeable loss of precision, although it is not as catastrophic as before.

In our finite difference computation, we run the risk of catastrophic loss of precision when we perform the subtraction f(x0+h)−f(x0)f(x_0+h) - f(x_0)f(x0​+h)−f(x0​). Because we take hhh to be small, these two numbers may be very close. Let's say the error in the subtraction is roughly ϵMf(x0)\epsilon_M f(x_0)ϵM​f(x0​). Then when we try to perform the finite difference approximation (1), what we are actually computing is more like

f′(x0)≈f(x0+h)−f(x0)+ϵMf(x0)h+O(h)=f(x0+h)−f(x0)h+f(x0)ϵMh+O(h).\begin{aligned} f'(x_0) &\approx \frac{f(x_0+h) - f(x_0) + \epsilon_M f(x_0)}{h} + \mathcal{O}(h) \\ &= \frac{f(x_0+h) - f(x_0)}{h} + f(x_0)\frac{\epsilon_M}{h} + \mathcal{O}(h). \end{aligned}f′(x0​)​≈hf(x0​+h)−f(x0​)+ϵM​f(x0​)​+O(h)=hf(x0​+h)−f(x0​)​+f(x0​)hϵM​​+O(h).​

The error in our approximation is

Error=f′(x0)−f(x0+h)−f(x0)h=f(x0)ϵMh+O(h), \textrm{Error} = f'(x_0) - \frac{f(x_0+h) - f(x_0) }{h} = f(x_0)\frac{\epsilon_M}{h} + \mathcal{O}(h), Error=f′(x0​)−hf(x0​+h)−f(x0​)​=f(x0​)hϵM​​+O(h),

where the f(x0)ϵM/hf(x_0)\epsilon_M/hf(x0​)ϵM​/h term is due to floating point errors, and the O(h)\mathcal{O}(h)O(h) is the truncation error in our approximation formula.

On the one hand, we want to make hhh small so that O(h)\mathcal{O}(h)O(h) is small. But on the other hand, the smaller we make hhh, the larger we make f(x0)ϵM/hf(x_0)\epsilon_M/hf(x0​)ϵM​/h. To minimize the error in our computation of f′(x0)f'(x_0)f′(x0​), we need to choose hhh which makes both terms as small as possible.

We don't know the coefficients in

O(h)=C1h+C2h2+C3h3+… . \mathcal{O}(h) = C_1 h + C_2 h^2 + C_3 h^3 + \dots. O(h)=C1​h+C2​h2+C3​h3+….

If we did know the coefficients, we would have a more accurate approximation formula. But, for h≪1h \ll 1h≪1, we can at least ignore the higher order terms C2h2,C3h3→0C_2 h^2, C_3 h^3 \rightarrow 0C2​h2,C3​h3→0 since they decrease much faster than C1hC_1 hC1​h as h→0h \rightarrow 0h→0. Our goal is to make f(x0)ϵM/h+C1hf(x_0)\epsilon_M/h + C_1 hf(x0​)ϵM​/h+C1​h as close to zero as possible. We don't know the value of C1C_1C1​ (if we did we could perform a higher order approximation). So instead, let's just ignore the constants f(x0)f(x_0)f(x0​) and C1C_1C1​, and set them both equal to 1 (equivalently, we are assuming that C1C_1C1​ is roughly the same size as f(x0)f(x_0)f(x0​)). These is a very hand-wavey decision, but without knowing C1C_1C1​ there is not much we can do.

Assuming ϵ,h>0\epsilon, h > 0ϵ,h>0, then we are trying to solve

min⁡hϵMh+h. \min_h \frac{\epsilon_M}{h} + h. hmin​hϵM​​+h.

We can take the derivative with respect to hhh and set the result equal to zero to find a critical point:

0=ddh(ϵh+h)=−ϵh2+10=−ϵ+h2h=ϵ\begin{aligned} 0 &= \frac{d}{dh} \left(\frac{\epsilon}{h} + h\right) = -\frac{\epsilon}{h^2} + 1 \\ 0 &= -\epsilon + h^2 \\ h &= \sqrt{\epsilon} \end{aligned}00h​=dhd​(hϵ​+h)=−h2ϵ​+1=−ϵ+h2=ϵ​​

Let's assume ϵ≈ϵM\epsilon \approx \epsilon_Mϵ≈ϵM​, where ϵM\epsilon_MϵM​ is the machine precision, as returned by eps(Float64). That is, ϵ≈10−16\epsilon \approx 10^{-16}ϵ≈10−16, giving us

hideal=10−8. h_{\textrm{ideal}} = 10^{-8}. hideal​=10−8.

Plugging hidealh_{\textrm{ideal}}hideal​ back into equation (2), we see that we expect the error to be on the order of ϵ/h+h≈10−8\epsilon/h + h \approx 10^{-8}ϵ/h+h≈10−8

Checking Results

These are all rough calculations, so let's check them with a numerical experiment. Let's compute f′(x0)f'(x_0)f′(x0​) using our finite difference approximation with several values of hhh, and check the relative error in the computed value compared to the analytic solution.

using CairoMakie
CairoMakie.set_theme!(CairoMakie.theme_latexfonts(); fontsize=14, Axis=(titlegap=10,))

f(x) = exp(2*x)
fprime_finite_diff(x, h) = (f(x+h) - f(x)) / h

x0 = 1.0
fprime_analytic = 2*exp(2*x0)

h_vals = [10.0 ^ i for i in -15:0.5:0]
fprime_finite_diff_vals = [fprime_finite_diff(x0, h) for h in h_vals]

fprime_finite_diff_errors = abs.(fprime_finite_diff_vals .- fprime_analytic)
fprime_finite_diff_relative_errors = fprime_finite_diff_errors ./ fprime_analytic

fig = Figure(size=(468, 350))
ax = Axis(fig[1, 1],
    xscale=log10, yscale=log10,
    xlabel=L"h", ylabel="Relative Error",
    title=L"Finite Difference Approximation of $f(x) = e^{2x}$",
    limits=(1e-15, 1e0, 1e-15, 1e0))
vlines!(ax, [1e-8], linewidth=2, linestyle=:dash, color=Makie.wong_colors()[2], label="h=1e-8")
hlines!(ax, [1e-8], linewidth=2, linestyle=:dash, color=Makie.wong_colors()[3], label="Error=1e-8")
lines!(ax, h_vals, fprime_finite_diff_relative_errors, linewidth=2, color=Makie.wong_colors()[1], label="Relative Error")
Legend(fig[2, 1], ax, framevisible=false, orientation=:horizontal)

Indeed, we see that the error is minimized around h=10−8h=10^{-8}h=10−8, and that the the relative error in our approximation at h=10−8h=10^{-8}h=10−8 is roughly 10−810^{-8}10−8, as we predicted. This is a big improvement over our result when choosing hhh to be machine precision, for which the relative error was over 10−110^{-1}10−1. In other words, we went from over 10% error to only 0.0000001% error.

In conclusion, by analyzing the effects of floating point errors in our computation, we have successfuly obtained a much more accurate finite difference approximation of f′(x0)f'(x_0)f′(x0​) than the one we obtained by simply taking hhh to be very small.

A Simpler Optimization Method

I used calculus to find a critical point of (3). If we wanted to be less fancy about finding the minimum, we could have simply plotted each of the terms in that equation on a log-log scale and found the point where they intersect.

g1(h) = 1e-16 / h
g2(h) = h

h_vals = [10.0 ^ i for i in -15:0.5:0]
g1_vals = [g1(h) for h in h_vals]
g2_vals = [g2(h) for h in h_vals]

fig = Figure(size=(468, 350))
ax = Axis(fig[1, 1],
    xscale=log10, yscale=log10,
    xlabel=L"h",
    title="Roundoff vs Approximation Errors", titlefont=:regular,
    limits=(1e-15, 1e0, 1e-15, 1e0))
lines!(ax, h_vals, g1_vals, linewidth=2, color=Makie.wong_colors()[1], label="ε / h (Roundoff Error)")
lines!(ax, h_vals, g2_vals, linewidth=2, color=Makie.wong_colors()[2], label="h (Approximation Error)")
Legend(fig[2, 1], ax, framevisible=false, orientation=:horizontal)

We can see that the lines intersect near h=10−8h=10^{-8}h=10−8, and that the roundoff and approximation terms both have a value around 10−810^{-8}10−8 (which will add up to 2×10−82 \times 10^{-8}2×10−8). This is consistent with our original prediction.

Higher Accuracy With Second Order Central Difference

What if we decide 10−810^{-8}10−8 relative precision is not enough? We can use a higher-order method to obtain a more accurate result, but then our ideal value for hhh will change. Let's consider the second-order central difference method

f′(x)=f(x+h)−f(x−h)2h+O(h2). f'(x) = \frac{f(x+h) - f(x-h)}{2h} + \mathcal{O}(h^2). f′(x)=2hf(x+h)−f(x−h)​+O(h2).

Accounting for roundoff errors and ignoring constant factors as we did before, we get

Error≈ϵMh+h2, \textrm{Error} \approx \frac{\epsilon_M}{h} + h^2, Error≈hϵM​​+h2,

which can be minimized by choosing

hideal=ϵM1/3≈10−5.3  ⟹  Error≈10−10.7. h_{\textrm{ideal}} = \epsilon_M^{1/3} \approx 10^{-5.3} \implies \textrm{Error} \approx 10^{-10.7}. hideal​=ϵM1/3​≈10−5.3⟹Error≈10−10.7.
f(x) = exp(2*x)
fprime_central_diff(x, h) = (f(x+h) - f(x-h)) / (2*h)

x0 = 1.0
fprime_analytic = 2*exp(2*x0)

h_vals = [10.0 ^ i for i in -15:0.5:0]
fprime_central_diff_vals = [fprime_central_diff(x0, h) for h in h_vals]

fprime_central_diff_errors = abs.(fprime_central_diff_vals .- fprime_analytic)
fprime_central_diff_relative_errors = fprime_central_diff_errors ./ fprime_analytic

fig = Figure(size=(468, 350))
ax = Axis(fig[1, 1],
    xscale=log10, yscale=log10,
    xlabel=L"h", ylabel="Relative Error",
    title=L"Central Difference Approximation of $f(x) = e^{2x}$",
    limits=(1e-15, 1e0, 1e-15, 1e0))
vlines!(ax, [10.0 ^ (-16/3)], linewidth=2, linestyle=:dash, color=Makie.wong_colors()[2], label="h = epsilon")
hlines!(ax, [10.0 ^ (-10.7)], linewidth=2, linestyle=:dash, color=Makie.wong_colors()[3], label="Error")
lines!(ax, h_vals, fprime_central_diff_relative_errors, linewidth=2, color=Makie.wong_colors()[1], label="Relative Error")
Legend(fig[2, 1], ax, framevisible=false, orientation=:horizontal)

We see that our predictions were correct, and we can now approximate f′(x0)f'(x_0)f′(x0​) to a relative precision of ≈1010.7\approx 10^{10.7}≈1010.7, which is more precision than what we could do with the first-order method.
